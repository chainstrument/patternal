## Intention

Le Factory Method définit une **interface pour créer un objet**, mais délègue aux sous-classes la décision de la classe concrète à instancier. Il permet d'éliminer le couplage entre le créateur et les produits concrets.

## Problème résolu

Imaginez un framework de logistique. Il gère des camions — mais un jour, on veut aussi gérer des bateaux. Sans Factory Method, le code est truffé de `if (type === 'truck') ... else if (type === 'ship') ...`. Chaque ajout d'un nouveau moyen de transport nécessite de modifier le code existant, violant le principe Open/Closed.

## Implémentation TypeScript

```typescript
// factory-method.ts
// Produit — interface commune
interface Transport {
  deliver(from: string, to: string): string
}

// Produits concrets
class Truck implements Transport {
  deliver(from: string, to: string): string {
    return `[CAMION] Livraison routière de ${from} à ${to}`
  }
}

class Ship implements Transport {
  deliver(from: string, to: string): string {
    return `[BATEAU] Livraison maritime de ${from} à ${to}`
  }
}

// Créateur — déclare la factory method
abstract class Logistics {
  abstract createTransport(): Transport

  planDelivery(from: string, to: string): string {
    const transport = this.createTransport()
    return transport.deliver(from, to)
  }
}

// Créateurs concrets
class TruckLogistics extends Logistics {
  createTransport(): Transport { return new Truck() }
}

class ShipLogistics extends Logistics {
  createTransport(): Transport { return new Ship() }
}

// Usage
function createLogistics(type: 'truck' | 'ship'): Logistics {
  if (type === 'truck') return new TruckLogistics()
  return new ShipLogistics()
}

const logistics = createLogistics('ship')
console.log(logistics.planDelivery('Paris', 'New York'))
// [BATEAU] Livraison maritime de Paris à New York
```

## Variante fonctionnelle

En TypeScript moderne, on peut exprimer le Factory Method avec de simples fonctions :

```typescript
// factory-method-functional.ts
type NotifierType = 'email' | 'sms' | 'push'

interface Notifier {
  send(message: string, recipient: string): void
}

const notifiers: Record<NotifierType, () => Notifier> = {
  email: () => ({ send: (msg, to) => console.log(`Email à ${to}: ${msg}`) }),
  sms:   () => ({ send: (msg, to) => console.log(`SMS à ${to}: ${msg}`) }),
  push:  () => ({ send: (msg, to) => console.log(`Push à ${to}: ${msg}`) }),
}

function createNotifier(type: NotifierType): Notifier {
  return notifiers[type]()
}

const notifier = createNotifier('email')
notifier.send('Votre commande est expédiée', 'user@example.com')
```

## Exemples dans le monde réel

- **`React.createElement`** — crée des éléments React sans connaître leur implémentation DOM réelle
- **`express.Router()`** — fabrique des routeurs avec la même interface quelque soit la configuration
- **`winston.createLogger()`** — crée des loggers avec des transports configurables
- **`pg.Pool`** — crée des connexions à la BDD selon la configuration

## Quand l'utiliser

- Le code qui crée des objets ne doit pas savoir quelle classe concrète instancier
- Les sous-classes doivent pouvoir changer le type d'objet créé
- Vous voulez fournir un hook d'extension aux utilisateurs de votre bibliothèque

<Callout variant="info" title="Factory Method vs Abstract Factory">
Le **Factory Method** crée **un seul type de produit** via l'héritage. L'**Abstract Factory** crée des **familles de produits liés** via la composition.
</Callout>
